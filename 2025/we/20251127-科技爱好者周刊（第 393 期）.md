## 第二波“开放 API”浪潮

https://brandur.org/second-wave-api-first

十五年前的2011年，云服务开始兴起，诞生了第一波“开放 API”的浪潮。

> Facebook：发布了 Open Graph API，为深入探索它的平台数据提供了极其强大的功能。
> 
> Twitter：API 几乎完全开放。甚至不需要 OAuth 令牌——只需用户名/密码就能进行身份验证，访问几乎所有内容。
> 
> GitHub：非常出色的 API 设计，开发者可以访问任何需要的功能，并且尝试了 API/HATEOAS 等具有前瞻性的想法。

当时的想法是，开放的 API 有助于用户和第三方，深入使用平台，添加插件和扩展程序，从而推动增长，提高产品留存率和用户满意度。

那时还出现了一个叫做 ProgrammableWeb 的网站（现已关闭），它的名字就代表了当时人们的信念，互联网应该是可以编程的，把各种数据拼接在一起。

但是，后来的发展跟人们的预想完全相反。

平台发现，开放 API 不仅无法盈利（没法做广告），反而会带来用户流失，自家的数据帮助了其他公司的生意。于是，它们纷纷改变做法，限制和关闭 API，开始将用户保留在自家的“围墙花园”里面。

> **FaceBook**：自从剑桥分析公司事件（窃取用户数据，用于广告投放）后，Graph API 的功能受到了极大的限制，接入需要通过严格的应用审核流程。
>
> Twitter：新功能不再提供 API，数据流的访问权限也逐渐被关闭。第三方客户端受到限制，最终被彻底禁用。
>
> Instagram：完全弃用了之前公开的 API，要么通过官方应用使用 Instagram，要么就完全不用。
>
> GitHub：虽然保持开放的 API，但也采取了一定程度的管控。使用必须经过身份验证，并且实施了严格的速率限制，通过 API 来构建完整的第三方应用程序变得很困难。

2025年的年中，事情开始出现变化，功能强大的大模型迅速崛起，成为程序员最有用的新工具，主导了所有的媒体讨论。

但是，如果大模型不能执行代码，用处就不大。它的最大价值来自自动化（比如龙虾），而不是内容生成，因为只有通过自动化，大模型才能大量用于企业和生产环境。

自动化需要大模型能够调用其他工具，这意味着，那些工具必须开放自己的 API。

突然之间，API 不再是负担，而成为重要的接入渠道，让各种 Agent 能够代表用户完成工作。尤其对于产品差异较小的市场，API 的可用性在不久的将来很可能成为决定产品胜败的关键。

同样两家银行，一家有 API，另一家没有。后者就很可能被市场淘汰，因为大模型无法接入它，也就无法自动化转账。

种种迹象表明，第二波“开放 API”浪潮正在到来。

各种平台开始意识到，谁尽快向 AI 开放 API，谁就将抢占先机。所以，腾讯才会在龙虾 OpenClaw 爆红以后，以最快的速度开放了微信的接口，让龙虾可以向微信发消息。

其他产品也类似，早先就纷纷通过 MCP 和 Skill，开放自家的操作接口。

新一波的开放浪潮，将远比上一波更声势浩大、更易用。

（1）这次开放的不仅是云服务，还有很多人们日常需要的生活服务：外卖、电商、银行……甚至还有很多意想不到的服务，比如大模型预订餐厅和球馆。

（2）不再需要编程，人们将使用自然语言输入，由大模型翻译后调用 API。

（3）API 的使用对象，以前是应用程序，现在是消费者本人。

（4）API 的使用目的，以前主要是获取数据，现在是代表用户行事。

（5）结论：随着龙虾类 Agent 产品和运作模式的日益普及，API 的可用性有望成为竞争优势，帮助提供商在竞争中脱颖而出。结果很可能是，开放 API 在全球范围内的爆炸式增长。

## 软件开发的第三种方式

1998年，埃里克·S·雷蒙德出版了经典著作《大教堂与集市》，提出软件开发有两种方式。

一种方式是大教堂（Cathedral），软件经过精心规划，由一支专业的开发团队负责开发和管理，代码通常是闭源的。

另一种方式是集市（bazaar），软件是开放的、透明的、社区驱动的。

现在又出现了第三种方式：[神秘屋](https://www.dbreunig.com/2026/03/26/winchester-mystery-house.html)。

![](https://cdn.beekka.com/blogimg/asset/202604/bg2026040502.webp)

神秘屋是美国加利福尼亚的一幢19世纪大宅，由温彻斯特夫人建造。

温彻斯特夫人热爱建筑学，就决定亲自建设。她设计每一间房间，亲自监工。有时候，房间造好后或者造到一半，她不满意，就会拆除重建。

这导致这幢建筑的每个房间风格都不一样，整个建筑的结构如同迷宫一般。根据当年的记录，重建次数最多的一个部分，拆毁重建了16次。由于不断扩建，一些房间的窗户和门重建时也没拆掉，而是直接砌进了墙里。房屋在不同时期加盖了多层，最高达到五层。它的建筑风格主要是维多利亚式，但有些部分又是哥特式和罗马式。

它大约有160个房间、2000扇门、10000扇窗户、47个楼梯、47个壁炉、13个浴室和6个厨房。1922年，温彻斯特夫人去世后对外开放，人们称它为“神秘屋”。

如今，很多程序员就是温彻斯特夫人。

他用 AI 开发软件，自己提出需求，想要什么就让 AI 开发什么，既没有代码审查，也没有代码测试，充分满足自己的个性。

这种软件的特点是风格独特，高度个性化，规模庞大，不断扩张。开发工作几乎总是累加式的，需要时添加代码，出现 bug 就修复，几乎没有动力去精简代码。而且，开发过程很有趣，会让开发者乐在其中。

缺点是通常缺乏文档，对局外人来说晦涩难懂，就像“神秘屋”一样。

1、[风力发电机的木质基座](https://www.bbc.com/news/science-environment-67718719)

目前，风力发电机的基座都是圆柱形钢塔。随着它的直径和高度的增长，运输和架设难度快速上升。

![](https://cdn.beekka.com/blogimg/asset/202312/bg2023123009.jpeg)

他们将多层木材粘合并压缩在一起，形成圆弧状。然后将这些部件带到工地，粘合成一个圆柱体，堆叠成塔。

![](https://cdn.beekka.com/blogimg/asset/202312/bg2023123010.webp)

这样的好处就是，基座的重量大大减轻，并且可以分块安装，使得运输和架设变得容易。

![](https://cdn.beekka.com/blogimg/asset/202312/bg2023123011.webp)

走进基座的内部，可以清晰地看到墙壁的木质纹理。

![](https://cdn.beekka.com/blogimg/asset/202312/bg2023123012.webp)

![](https://cdn.beekka.com/blogimg/asset/202312/bg2023123013.webp)

如果这种木板被证实，可以胜任风力发电机的基座，具有足够的强度，并且能够防火。那么，它就也可以用来建造高层建筑，具有广阔的前景。

## 文摘

1、[为什么肥皂有清洁作用？](https://theconversation.com/how-does-soap-keep-you-clean-a-chemist-explains-the-science-of-soap-247559)

各种物质对水的亲合程度是不一样的。

化学家将被水吸引的分子归类为亲水分子，即喜欢水的分子。亲水分子可以溶解于水。

可想而知，洗手的时候，亲水物质会被流水冲走。

另一类分子被化学家称为疏水分子，也就是说它们怕水。疏水分子不溶于水。

油就是疏水物质，油和水根本无法混合。因此，洗手的时候，仅靠水无法去除油脂。

这时肥皂就派上用场了。

肥皂是一种复杂的分子，既亲水又怕水。具体来说，肥皂分子形似蝌蚪，头部圆，尾部长；头部亲水，尾部疏水。这是肥皂具有超强清洁能力的原因。

![](https://cdn.beekka.com/blogimg/asset/202504/bg2025043005.webp)

洗手的时候，肥皂分子聚集在一起，包裹住手上的污垢，形成所谓的胶束结构。

![](https://cdn.beekka.com/blogimg/asset/202504/bg2025043006.webp)

从分子层面来看，它看起来就像一个气泡，包裹着疏水性的污垢。肥皂分子的亲水“头”位于表面，而疏水“尾”位于胶束内部。这种结构可以锁住污垢，然后用流水将其全部冲走。

为了达到最佳效果，请在水槽边洗手至少 20秒。搓手有助于迫使肥皂分子渗入污垢，将其分解并包裹起来。

1、[合同软件开发的糟糕现状](https://smustafa.blog/2025/04/30/the-abysmal-state-of-contract-software-development/)

有些程序员是基于项目的合同工，不是正式的雇员。

这些程序员选择合同工，而不是稳定的工作，是因为想要灵活性和短期经济利益。灵活性指的是，工作时间可以自己安排，而且你可以同时签订多份合同。

可惜的是，现实情况是，公司雇佣了大量合同工，他们没有福利，解雇起来也容易得多，而且工资比全职员工低。

我知道这些，因为我干过好几次合同工。

除了薪酬和福利不如比全职员工，你还根本没有带薪休假。如果生病了或者需要休息一天，就根本拿不到工资。

合同工还有一个问题就是，他们被告知的工作和最终实际分配的工作之间，往往存在严重差异。

我曾经面试了一个 Java 的后端职位，但实际情况是，我几乎没有编写或维护任何 Java 代码，而是被要求几乎只写 React 代码，修复从另一个团队继承下来的有问题的 Jest 测试，以及极其缓慢的 Webpack 配置。

两个月后，我被解雇，理由是毫无根据的“绩效原因”。我知道这只借口，我遇到了太多自己根本无法控制的问题。

我的另一次合同工经历，也是如此。我在团队里轮班待命，但周六早上值班却没有工资；我提交的工时表被断然拒绝，老板打电话问我为什么要加班。

后来我发现，我的雇主不愿意支付我加班费，后来我被解除了合同，他们在电话里告诉我不胜任这项工作。

总之，现在的软件合同工有各种弊端，却得不到任何好处。如果有人能从合同工变成全职员工，那当然很好，但在我工作过的每家公司里，合同工都是二等公民。
