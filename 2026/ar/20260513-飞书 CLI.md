# 当办公套件遇上 AI Agent：开源项目飞书 CLI 介绍

## 一、引言 

今年春节前，龙虾 OpenClaw 突然爆火，引发各种云服务纷纷开放 API 让它接入。

有一类非常重要的软件，也在这期间开放了 API，但是没有得到外界的太多关注。那就是办公套件。

![](https://cdn.beekka.com/blogimg/asset/202605/bg2026051409.webp)

**办公套件开放 API**，意义不仅在于办公任务自动化，更在于软件开发者可以通过它接触到企业软件市场，因为办公套件的主要用户是企业，为他们定制 AI 生产力工具，有着巨大的市场空间。

下面就是前几个月推出命令行 API SDK 的办公套件名单。

- [Google WorkSpace](https://github.com/googleworkspace/cli)
- [飞书](https://github.com/larksuite/cli)
- [钉钉](https://github.com/DingTalk-Real-AI/dingtalk-workspace-cli)
- [企业微信](https://github.com/DingTalk-Real-AI/WecomTeam/wecom-cli)

上面这四家办公套件，现在都可以在命令行调用自家 API，完成各种操作。其中，谷歌 WorkSpace API 是半官方的，是他们的一个工程师个人在维护，其他三家都是官方的。

三家国内办公套件之中，[飞书 CLI](https://github.com/larksuite/cli) 规模最大，开发最活跃，影响力最广。一个证明是，它的 GitHub Star 最近超过了10000。

![](https://cdn.beekka.com/blogimg/asset/202605/bg2026051401.webp)

今天，我结合自己的使用体会，简单介绍一下这个项目，大家可以看到，**借助 AI 操作飞书**，真的非常实用。

## 二、MCP 还是 CLI

首先要区分一下，工具软件向 AI 开放自身能力有两种方式：MCP 和 CLI。

![](https://cdn.beekka.com/blogimg/asset/202605/bg2026051410.webp)

大多数工具采用 MCP 方式，比如 Notion / Monday / Figma，这样暴露出来的接口比较少，只供 AI 调用，可控程度高，而且随时可以更换，对服务提供商更简单、更方便。

CLI 是更彻底、更可靠的方式，不仅 AI 可以用，外部程序员也可以手工调用。官方必须给出完整的 API 接口定义，而且要保持稳定，不能随意更换。

所以，如果是严肃用途，都应该**优先采用官方的 CLI SDK**。前一节列出的四家办公套件，都属于这一类。

## 三、飞书 CLI 的基本情况

[飞书 CLI](https://github.com/larksuite/cli) 是3月28日上线，开发的活跃程度堪比龙虾。迄今不过40多天，就发了30个版本，几乎每个工作日都发新版本。

它的功能相当全，现有功能几乎都提供 API，已经支持飞书的100多项能力。我刚看到它的自述文件时都惊了，居然开放了这么多 API。

![](https://cdn.beekka.com/blogimg/asset/202605/bg2026051402.webp)

我数了一下，一共17个业务域，囊括了企业办公的各方面，现在都能让 AI 自动化，定制出各种工作流。

这些 API 都是开箱即用，你可以直接使用某一种能力，也可以深度编排多种能力，封装和扩展飞书的基础设施。

这个项目既是飞书官方的重点开源项目，也是 GitHub 的热门项目，外部参与度也非常高。因为飞书有国际版，国内外开发者都有参与，fork 和 issue 非常多，我看有不少外部 PR 被合并进主干了。

## 四、项目结构和 Skill 封装

飞书 CLI 项目是多层封装结构。

最底层是[飞书 OpenAPI 定义](https://www.feishu.cn/content/785259988660)，给出了每一个具体的 API 接口，包括参数、返回值、错误类型等等。

第二层是 API 命令层，每一个命令与 OpenAPI 一一映射。

第三层是快捷命令层，提供智能默认参数，简化调用写法，输出便于阅读的格式，尽量对用户友好，降低使用门槛。

在第三层的架构之上，它又为 Agent 封装了 Skill（预置的能力扩展包），大大方便了 AI 的调用，保证操作正确，防止出现错误的参数使用。

内置的 Skill 现在有24个，基本都是官方认定的最佳实现。

![](https://cdn.beekka.com/blogimg/asset/202605/bg2026051403.webp)

## 六、典型能力

官方有一个[《飞书 CLI 的 100 多个新本领》](https://bytedance.larkoffice.com/wiki/CLNjwBozvi11IjkeChOcltHinye)的能力清单，大家可以看一下，你可以用这个库做什么事情。

（1）文档功能。我平时写的是 Markdown 文档，但飞书是可视化编辑器，就提交 Markdown 原稿让它进行排版，省得自己粘贴和发布。

> 把这篇 Markdown 内容创建成飞书文档，排版要好看。

飞书支持文档评论，一般都是修改意见，可以让 AI 根据评论自动修改文档。

> {{文档链接}} 根据评论修改文档，修改后，用划词评论标识出修改点。

你也可以让 AI 生成评审意见。

> {{文档链接}}阅读一下这篇文档，看下作为提供给外部用户的说明文档，是否足够清晰、简洁。不要直接改文档，只把你觉得有优化空间的位置划词评论出来，在评论里写上修改意见。

（2）画板功能。直接根据文档内容，生成可编辑的架构图、流程图插入文档。

![](https://cdn.beekka.com/blogimg/asset/202605/bg2026051407.webp)

（3）幻灯片：将飞书文档转成飞书幻灯片。

![](https://cdn.beekka.com/blogimg/asset/202605/bg2026051408.webp)

（4）工作流。编排一个工作流，跨多个功能域，比如获取某个群聊的参与者名单 --> 参看每个参与者的日历，找到空闲时间 --> 为这些人约一个在线会议。

> 帮我看一下【XX】群里所有人的日历，然后下周找一个大家都合适的时间开一小时的讨论会

或者，让 AI 读取邮箱，一旦有重要的未读邮件，就转发到讨论群。

> 查看我所有的未读邮件，重要邮件发摘要到【XX】项目群

## 飞书CLI的技术优势

- 架构上，飞书 CLI 设计了「三层命令架构」，真正做到了Agent友好：

第一层 Shortcut 快捷命令：面向人和 AI 共同优化，提供智能默认参数、友好输出和 dry-run（试运行） 预览，降低调用门槛。

第二层 API Commands：与飞书 OpenAPI 一一映射，同时具备动态发现能力。

第三层 Raw API Calls —— 即便在任何老版本 CLI 中依然可以访问所有飞书最新开放的 OpenAPI 能力，理论上可以做到无限扩展。这一层也可以让模型充分发挥智能：当 Agent 遇到复杂任务或预设命令覆盖不到的场景时，可以直接通过对 Raw API 的理解，自己尝试找到各种可能的解决方案。

- 在命令架构之上，飞书CLI还提供了 Skills —— 给 Agent 预制的能力扩展包。每个 Skill 封装了使用飞书 CLI 完成某类典型场景任务的最佳路径。这解决的是一个很现实的问题：开放 API 的数量多，并不等于 AI 用得好。Agent 面对几百个命令，往往不知道怎么组合才最顺。Skills 把「怎么正确使用」这些 API 的方法沉淀成 Skill。Agent 不用在每次任务里重新摸索最佳实践，而是直接站在飞书已有的积累上，让 AI 从「能用」跨到「好用」。

社区每天真实的 Issue 和 PR，让产品每天都在被真实场景打磨。开源迭代的好处是收敛到正确的速度，以及开源开发者们的开发速度加成，这在 AI 时代是压倒性的优势。
