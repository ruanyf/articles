# 我们离理想大模型还有多远：MiniMax M3 解读

## 一、引言

我问大家一个问题，对于普通用户，理想的大模型应该什么样？

在我看来，日常使用的大模型，最好同时满足四个条件。

（1）性能强，尤其具有强大的编程能力。

（2）上下文窗口大，最好达到100万 Token。

（3）多模态，不仅处理文本，还处理图片、视频、音频等，最好还能操作计算机。

（4）价格不贵，量大，便宜，管饱。

如果满足这四点，一个模型就够了，不用频繁切换模型。后期计算使用量和费用，也变得简单。

那么，有没有模型满足这些条件呢？

据我所知，好像没有。国外的旗舰模型太贵，国内的开源模型往往是单模态，图像、视频、音频都有各自的模型，而且上下文窗口也不大。

## 二、MiniMax M3

本周一，MiniMax 发布了他们最新的[旗舰模型 M3](https://www.minimaxi.com/models/text/m3)。

我原以为只是一次常规的版本升级，但是读了[发布报告](https://www.minimaxi.com/blog/minimax-m3)，震惊地发现，**它竟然同时符合上面的四个条件**。

相比上一个版本 M2.7，上下文窗口从20万增大到100万，从单模态的文本模型升级为多模态，基准测试的得分非常高，并且有配套的 Token Plan。

看上去，它完全是理想大模型的合格候选人。这激发了我的浓厚兴趣，下面就是一些试用体会。

## 三、100万上下文窗口

先解释一下，为什么较大的上下文窗口比较好。

上下文窗口（context）是模型一次能够处理的最大 Token 数量。这个窗口越大，能够向模型提供的背景信息就越多，理论上生成的结果也会越准确。

由于每一次模型交互，都会附带本轮对话已经完成的步骤，所以**大窗口对于大型项目或长程操作非常重要**。

开源模型的上下文窗口，一般都是20万 Token。为什么不做得更大一些呢？原因是 Token 越多，计算量越大。每一个 Token 都要计算跟其他 Token 的关系，随着上下文窗口的增大，计算量会急剧增长，从而影响模型的性能和成本。

MiniMax M3 这一次把上下文窗口做到了100万 Token，属于普通用户能够接触到的最大级别。

它敢做这么大，是因为使用了新的技术方案：稀疏注意力架构 MSA。这种方法将 Token 进行分块和筛选，选出代表性的 Token 参与计算，从而避免了全量计算。官方[介绍](https://www.minimaxi.com/blog/minimax-m3)说，计算量仅为“上代模型的 1/20”。

> “在100万上下文下，M3 每 token 计算量仅为上代模型的 1/20。在 prefilling 阶段，我们实现了超过 9 倍的加速倍率，在 decoding 阶段有超过 15 倍的加速优势。而且在多个对照实验中，MSA 的绝大部分能力与全注意力打平。”

这就是说，跟上一代相比，虽然上下文窗口增大了5倍，但是计算量反而是下降的，而且不影响生成效果（“绝大部分能力与全注意力打平”）。

## 四、多模态能力

MiniMax M3 支持图片和视频的输入，并能操作电脑桌面。

多模态测试集 OmniDocBench 上，MiniMax M3 得分超过 Gemini 3.1 Pro，在面向自主 Agent 的端到端评测框架 Claw-Eval 上，MiniMax M3 得到最高分。

## 五、MiniMax Code 和 Token Plan

MiniMax Code 是对标 Claude Code/Codex 的 Agent 产品，具备 Computer Use 能力。

官方示例：用户可以在手机上说：“帮我打开本地 ERP 客户端，按这份 Excel 批量录入发票信息”。

MiniMax Code 会自动在电脑端完成跨应用、跨文件、跨系统的操作。基于社区出色的开源项目 OpenCode 和 Pi Agent 构建，针对 M3 一起进行训练。

它有配套的 Token Plan，价格在国内属于最低的那一档。

> - Plus ¥49 / 月：6 亿 token ≈ Claude Pro - $20 月度容量的 5 倍
> - Max ¥119 / 月：18 亿 token ≈ Claude Max 5x $100 月度容量的 2 倍
> - Ultra ¥469 / 月：55 亿 token ≈ Claude Max 20x $200 月度容量的 3 倍

如果按照相同价格算，约是 Claude 订阅的 15 倍用量。
