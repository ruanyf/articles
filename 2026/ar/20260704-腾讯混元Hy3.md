# 混元大模型三年一剑，姚顺雨腾讯首秀

## 一、引言

说起混元大模型，大家想到什么？

![](https://cdn.beekka.com/blogimg/asset/202607/bg2026070502.webp)

混元是腾讯自研的旗舰模型，早在2023年9月就发布了，可是后来渐渐失去了话题性。去年12月，混元 2.0 发布，也没有引起太多关注。

现在，混元 3.0 来了。这一次，腾讯显然寄予厚望，4月份就公开了预览版 [Hy3-preview](https://www.tencent.com/zh-cn/articles/2202320.html)，明确表示编码能力和 Agent 能力有大幅提升，并进行了大量推广，现在还有两周的免费试用。

![](https://cdn.beekka.com/blogimg/asset/202607/bg2026070501.webp)

背后的一个原因是，这个模型是姚顺雨离开 OpenAI、担任腾讯首席 AI 科学家后首个亮相的成果，组织人员和模型架构都进行了深度调整，跟前两代完全不一样。

那么，混元 3.0 模型到底强不强，三年后能不能重回一线，我就来测一下。

## 二、模型定位

混元 3.0 的评分还没有公布，单看参数，它不是一个特别大的模型，属于混合专家模型（MoE），总参数 295B，运行时激活的参数 21B，上下文长度 256K。

这些参数表明，它内部的计算量不算大，跟那些运算规模极大的旗舰模型肯定有差距。

我选了两个规模类似的国产模型进行后面的对比：GLM 5.1 和 DeepSeek 4 flash。

![](https://cdn.beekka.com/blogimg/asset/202607/bg2026070505.webp)

从参数看，GLM 5.1 的运算量最大，混元 3.0 其次，最后是 DeepSeek 4 Flash，但它的上下文长度最大，可以处理更多的 Token。

实际测试也是如此，DeepSeek 4 Flash 生成速度最快，混元 3.0 其次，GLM 5.1 最慢。

由此可见，混元 3.0 的设计思路不是追求性能，而是追求性能和成本/运算速度的平衡。

## 三、BridgeBench 测试基准

目前，最热门的测试基准大概是 [BridgeBench](https://www.bridgebench.ai/)，号称“全世界排名第一的氛围编程测试基准”。

![](https://cdn.beekka.com/blogimg/asset/202607/bg2026070506.webp)

![](https://cdn.beekka.com/blogimg/asset/202607/bg2026070507.webp)

它一共170余道题，专门测试模型的代码能力。

它的运行方法和评分方法，都没有公开，但是[部分测试题](https://www.bridgebench.ai/test-prompts)是公开的。我选择了一些题目，三个模型都跑一下，看谁的表现最好。

不过，这些题目都很大很复杂，跑起来太费时间，有时一道题要跑一个小时。简单的网页 UI 测试，我就用[自己收集的测试题](https://github.com/ruanyf/ai-test-case)了。

另外，测试工具是腾讯的 WorkBuddy，桌面的 AI 工作台。

![](https://cdn.beekka.com/blogimg/asset/202607/bg2026070508.webp)

使用它的主要原因是，全量内置了混元 3.0，而且发布后的两周免费使用。

另外，内置模型中也包括了 GLM 5.1 和 DeepSeek 4 Flash。测试的时候，只要在它里面切换模型即可。

## 九、总结

经过这些测试，混元 3.0 的表现让人耳目一新，简单的任务不成问题，复杂的任务也能做出让人接受的结果。

它的表现明显好于 DeepSeek 4 Flash，很多时候也好于 GLM 5.1，至少不比它差。

考虑它的运行速度和成本，确实是一个性价比突出的模型，各方面比较均衡，可以作为日常主力的编程/Agent 模型使用。

Agent / Coding

Agent方向能力，验证模型在真实工程任务中的表现。

Hy3 正式版是一个快慢思考融合的 MoE 语言模型，总参数 295B、激活参数 21B，最大支持 256K 上下文长度，整体性能达到同尺寸模型最佳水平，特别适用于 Coding 与智能体（Agent）类场景。

Coding Agent 任务执行与生产力场景跃升以及复杂推理和数理能力的优化，

- Coding Agent 能力大幅升级（前端深度优化）：针对前端开发场景进行了深度优化，在前端需求理解、组件生成、交互逻辑编写及自动化 Debug 方面的可用性显著提升，旨在打造更流畅的开发闭环。
- Working Agent 通用办公能力全面拓宽：强化了面对复杂工作流时的任务规划、长文本信息处理与多工具调用能力，进一步贴近实际业务场景的自动化需求。
- Reasoning / STEM 复杂推理与数理能力显著跃升，突破 Agentic Reasoning瓶颈：充分释放了思考和工具使用的潜力，模型在多步骤逻辑推演、复杂数学计算及科学问题求解上的表现得到大幅强化。凭借更强的长链路规划、动态反思与使用外部工具进行数据分析、科学研究，模型能精准拆解并攻克复杂问题，为高阶数据分析、科研等硬核任务提供更可靠的智力。
